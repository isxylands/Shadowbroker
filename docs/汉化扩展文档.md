# ShadowBroker 汉化与数据源扩展文档

本文记录 ShadowBroker 的数据源来源、前后端数据流、以及添加新数据源时的推荐路径。重点服务于后续汉化：不仅翻译 UI，也要处理静态数据、外部 feed、运行时抓取结果和图层标签。

## 1. 数据源从哪里来

ShadowBroker 的数据源不是集中在一个配置文件里，而是由多套机制共同组成。

### 1.1 主仪表盘数据源

主仪表盘数据由后端抓取器写入统一的内存仓库：

- 数据仓库：`backend/services/fetchers/_store.py`
- 核心对象：`latest_data`
- 图层开关：`active_layers`
- 调度入口：`backend/services/data_fetcher.py`

`data_fetcher.py` 将数据更新分成两类：

- `update_fast_data()`：快数据，包含飞机、船舶、卫星、SIGINT、火车等。
- `update_slow_data()`：慢数据，包含新闻、GDELT、CCTV、数据中心、军事基地、电厂、天气、空气质量等。

后端启动后还会通过 APScheduler 周期性刷新这些 fetcher。

### 1.2 静态内置数据

静态数据主要放在：

```text
backend/data/
```

典型文件包括：

- `datacenters_geocoded.json`：数据中心
- `military_bases.json`：军事基地
- `power_plants.json`：电厂
- `tracked_names.json`：重点跟踪对象名称
- `plane_alert_db.json`：飞机告警/识别数据库
- `yacht_alert_db.json`：游艇告警/识别数据库
- `plan_ccg_vessels.json`：PLAN/CCG 船只识别数据

这些静态数据的读取逻辑通常在 `backend/services/fetchers/*.py` 中，例如：

- `backend/services/fetchers/infrastructure.py`
  - `fetch_datacenters()`
  - `fetch_military_bases()`
  - `fetch_power_plants()`

### 1.3 外部 API 与在线抓取

项目对外部数据源的归属说明集中在：

- `DATA-ATTRIBUTION.md`
- `README.md`

主要来源包括：

- OpenSky Network：商业和私人飞机
- adsb.lol：ADS-B 飞机数据、军机和航线补充
- aisstream.io：AIS 船舶位置流
- CelesTrak：卫星 TLE 数据
- USGS：地震
- NASA FIRMS：火点/热异常
- NOAA SWPC：空间天气
- GDELT：全球冲突/事件
- DeepState Map：乌克兰前线 GeoJSON
- OpenAQ：空气质量
- WRI Global Power Plant Database：全球电厂
- OpenMHZ：警消无线电 scanner feeds
- Meshtastic / APRS：无线电/网状网络节点

这些源大多由 `backend/services/fetchers/` 下的模块封装。

### 1.4 新闻 RSS 数据源

新闻源配置在：

```text
backend/config/news_feeds.json
```

读取和校验逻辑在：

```text
backend/services/news_feed_config.py
```

对应管理接口在：

```text
backend/routers/admin.py
```

主要接口：

- `GET /api/settings/news-feeds`
- `PUT /api/settings/news-feeds`
- `POST /api/settings/news-feeds/reset`

前端 Settings 面板已有 News Feeds 标签页，可以直接添加、修改、重置 RSS/Atom 源。

### 1.5 AI Intel 自定义 Pin Layer / Feed

这是最适合快速添加本地化数据源的机制。

相关文件：

- `backend/services/ai_pin_store.py`
- `backend/services/feed_ingester.py`
- `backend/routers/ai_intel.py`
- `frontend/src/components/AIIntelPanel.tsx`
- `frontend/src/lib/aiIntelClient.ts`

AI Intel 支持创建自定义图层，每个图层可以带一个 `feed_url`。后台 `feed_ingester.py` 会定期轮询这些 feed，将结果转成地图 pin。

feed 支持两类格式：

1. GeoJSON FeatureCollection
2. JSON pin 数组

每次刷新会替换该图层下的旧 pin。

## 2. 前后端数据流

前端主要通过三个接口拿仪表盘数据：

- `GET /api/bootstrap/critical`
- `GET /api/live-data/fast`
- `GET /api/live-data/slow`

这些接口定义在：

```text
backend/routers/data.py
```

前端轮询逻辑在：

```text
frontend/src/hooks/useDataPolling.ts
```

数据合并到浏览器端 store 的逻辑在：

```text
frontend/src/hooks/useDataStore.ts
```

主流程为：

```text
fetcher -> latest_data -> /api/live-data/* -> useDataPolling -> mergeData -> map/components
```

AI Intel pin layer 是另一条链路：

```text
feed_url -> feed_ingester -> ai_pin_store -> /api/ai/pins/geojson -> MaplibreViewer
```

## 3. 如何添加新的数据源

根据目标不同，有三条推荐路径。

### 3.1 只添加新闻源

适用场景：

- 添加中文新闻 RSS
- 替换英文新闻源
- 增加区域性中文媒体源

方式一：前端 Settings 面板添加。

方式二：直接编辑：

```text
backend/config/news_feeds.json
```

格式示例：

```json
{
  "name": "Example CN",
  "url": "https://example.com/rss.xml",
  "weight": 3
}
```

`weight` 范围是 1 到 5，权重越高，在新闻聚合、排序、风险评估中的影响越大。

### 3.2 添加地图点位数据源

适用场景：

- 中文本地事件点位
- 自定义基础设施点位
- 本地机构/设施/风险点数据
- 外部服务可以输出 GeoJSON 或 JSON

推荐使用 AI Intel 自定义图层。

创建图层时填写：

- 图层名称
- 可选描述
- 可选颜色
- `feed_url`
- `feed_interval`

GeoJSON feed 示例：

```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Point",
        "coordinates": [116.4074, 39.9042]
      },
      "properties": {
        "name": "示例点位",
        "category": "infrastructure",
        "description": "中文说明",
        "source_url": "https://example.com"
      }
    }
  ]
}
```

普通 JSON pin 数组示例：

```json
[
  {
    "lat": 39.9042,
    "lng": 116.4074,
    "label": "示例点位",
    "category": "infrastructure",
    "description": "中文说明",
    "source_url": "https://example.com",
    "confidence": 1.0
  }
]
```

支持的 pin 分类定义在 `backend/services/ai_pin_store.py`，包括：

- `threat`
- `news`
- `geolocation`
- `custom`
- `anomaly`
- `military`
- `maritime`
- `flight`
- `infrastructure`
- `weather`
- `sigint`
- `prediction`
- `research`

### 3.3 添加正式内置图层

适用场景：

- 需要出现在左侧 Data Layers 里
- 需要参与时间机器、AI 查询、相关性分析或主地图图层
- 数据结构复杂，不能只作为 pin 展示
- 需要后端定时抓取、缓存、鉴权或环境变量

基本步骤：

1. 新建或修改 fetcher

   ```text
   backend/services/fetchers/<your_source>.py
   ```

2. 在 `backend/services/fetchers/_store.py` 中添加数据 key

   - `DashboardData` 类型
   - `latest_data` 初始值
   - 如需图层开关，添加到 `active_layers`

3. 在 `backend/services/data_fetcher.py` 中注册调度

   - 高频数据加入 `update_fast_data()`
   - 低频数据加入 `update_slow_data()`
   - 特殊周期任务加入 `start_scheduler()`

4. 在 `backend/routers/data.py` 中加入 API 响应

   - `/api/bootstrap/critical`
   - `/api/live-data/fast`
   - `/api/live-data/slow`

5. 在前端类型中添加字段

   ```text
   frontend/src/types/dashboard.ts
   ```

6. 添加前端图层开关

   - `ActiveLayers`
   - `frontend/src/app/page.tsx`
   - `frontend/src/components/FilterPanel.tsx`

7. 添加地图渲染逻辑

   常见位置：

   ```text
   frontend/src/components/MaplibreViewer.tsx
   frontend/src/components/map/geoJSONBuilders.ts
   ```

8. 如数据来自第三方，补充 attribution

   ```text
   DATA-ATTRIBUTION.md
   ```

## 4. 汉化策略

ShadowBroker 已经有 UI 国际化目录：

```text
frontend/src/i18n/
```

但这只解决界面文案。数据源汉化需要分层处理。

### 4.1 UI 文案

翻译文件：

```text
frontend/src/i18n/translations/zh-CN.json
```

适合处理：

- 按钮
- 面板标题
- 图层名
- 提示语
- 设置项
- 快捷键说明

### 4.2 静态数据

对 `backend/data/*.json` 中的静态对象，不建议直接覆盖英文原字段。推荐增加中文显示字段。

示例：

```json
{
  "name": "Kadena Air Base",
  "name_zh": "嘉手纳空军基地",
  "country": "Japan",
  "country_zh": "日本",
  "operator": "United States Air Force",
  "operator_zh": "美国空军",
  "lat": 26.351,
  "lng": 127.767
}
```

优点：

- 保留原始数据，减少测试和匹配逻辑破坏
- 前端可以按语言选择显示字段
- 后续可逐步补齐中文字段

### 4.3 实时外部数据

对实时数据，例如新闻、AIS、ADS-B、GDELT、OpenAQ，不建议直接改写原始字段。

推荐做法：

- 保留 `title`、`name`、`source` 等原始字段
- 增加 `display_title_zh`、`display_name_zh`、`category_zh`
- 或在前端维护枚举/分类映射

### 4.4 新闻汉化

新闻层最适合通过中文 RSS 源实现本地化，而不是强行机器翻译英文源。

推荐优先级：

1. 添加中文 RSS 源
2. 保留部分国际英文源作为交叉验证
3. 如需翻译英文新闻，新增显示字段，不覆盖原文

### 4.5 枚举和分类汉化

大量字段是枚举，例如：

- `military_flight`
- `private_jet`
- `infrastructure`
- `weather`
- `threat`
- `high`
- `medium`
- `low`

这类字段适合在前端统一映射为中文标签，不应改动后端原始 key。

示例：

```ts
const categoryLabelsZh = {
  threat: '威胁',
  news: '新闻',
  infrastructure: '基础设施',
  weather: '天气',
  military: '军事',
};
```

## 5. 推荐落地顺序

建议按风险从低到高推进。

1. 补全 UI 汉化

   修改 `frontend/src/i18n/translations/zh-CN.json`。

2. 添加中文新闻源

   修改 Settings 中的 News Feeds，或编辑 `backend/config/news_feeds.json`。

3. 用 AI Intel feed_url 做中文点位数据原型

   先验证数据结构、显示效果、刷新频率和地图体验。

4. 汉化静态数据

   为 `backend/data/*.json` 增加中文显示字段，不覆盖原始字段。

5. 增加前端中文显示选择逻辑

   按当前语言优先显示 `*_zh` 字段，否则回退英文原字段。

6. 将稳定的数据源沉淀为正式内置图层

   按完整 fetcher -> router -> frontend layer 流程接入。

## 7. 具体数据源评估：CVE.org 与 Palo Alto Security Advisories

本节针对两个候选安全数据源给出推荐接入方式。

### 7.1 Palo Alto Networks Security Advisories

源地址：

```text
https://security.paloaltonetworks.com/rss.xml
```

验证结果：

- 这是可直接读取的 RSS 2.0 feed。
- feed item 标题中包含 CVE 编号、产品、漏洞标题和严重性。
- item link 指向 Palo Alto 官方漏洞公告详情页。

示例标题形态：

```text
CVE-2026-0257 PAN-OS: GlobalProtect Authentication Bypass Vulnerabilities (Severity: HIGH)
```

推荐落点：

```text
backend/config/news_feeds.json
```

原因：

- 项目已有 RSS/Atom 新闻聚合机制。
- 该 feed 是公告流，不是地理点位数据，不适合直接做地图 pin。
- 作为安全公告源，先进入 Threat Intel / News 面板最省改动。

添加方式：

```json
{
  "name": "Palo Alto Security",
  "url": "https://security.paloaltonetworks.com/rss.xml",
  "weight": 4
}
```

可以通过两种方式添加：

1. 前端 Settings 面板的 News Feeds 标签页添加。
2. 直接编辑 `backend/config/news_feeds.json`，然后等待下一轮 news refresh，或调用 `/api/refresh`。

注意事项：

- 现有 `fetch_news()` 每个 feed 只取前 5 条，适合公告流。
- 现有风险关键词对 cyber/security 不够敏感。若希望安全公告排序更靠前，可在 `backend/services/fetchers/news.py` 的 `risk_keywords` 中增加：

```python
"cve", "vulnerability", "exploit", "rce", "authentication bypass",
"privilege escalation", "command injection", "buffer overflow",
"zero-day", "critical", "severity"
```

### 7.2 CVE.org

源地址：

```text
https://www.cve.org/
```

评估结论：

- `https://www.cve.org/` 本身更像门户站点，不是一个可直接添加到 `news_feeds.json` 的 RSS 源。
- CVE 数据更结构化，适合使用 CVE Services API 或 CVEProject/cvelistV5 数据，而不是当作普通新闻源处理。

可用结构化来源：

```text
https://cveawg.mitre.org/api/cve/<CVE-ID>
https://cveawg.mitre.org/api/cve-id/<CVE-ID>
https://github.com/CVEProject/cvelistV5
```

示例：

```text
https://cveawg.mitre.org/api/cve/CVE-2026-0257
```

推荐落点：

短期不建议直接添加到现有 News Feeds。更合适的方案是新增一个正式“漏洞情报”数据层，例如：

```text
backend/services/fetchers/vulnerabilities.py
```

建议输出 key：

```text
latest_data["vulnerabilities"]
```

建议数据结构：

```json
{
  "id": "CVE-2026-0257",
  "title": "PAN-OS: GlobalProtect Authentication Bypass Vulnerabilities",
  "assigner": "palo_alto",
  "date_published": "2026-05-13T18:15:10.172Z",
  "date_updated": "2026-05-30T03:55:17.629Z",
  "severity": "HIGH",
  "cvss": 8.1,
  "cwe": ["CWE-565"],
  "vendors": ["Palo Alto Networks"],
  "products": ["PAN-OS", "GlobalProtect"],
  "references": ["https://security.paloaltonetworks.com/CVE-2026-0257"],
  "summary": "English source summary",
  "summary_zh": "中文摘要",
  "source": "cve.org"
}
```

完整内置图层接入步骤：

1. 新增 fetcher：

   ```text
   backend/services/fetchers/vulnerabilities.py
   ```

2. 在 `backend/services/fetchers/_store.py` 中增加：

   ```python
   vulnerabilities: List[Dict[str, Any]]
   latest_data["vulnerabilities"] = []
   active_layers["vulnerabilities"] = True
   ```

3. 在 `backend/services/data_fetcher.py` 中注册到慢数据层：

   ```python
   from services.fetchers.vulnerabilities import fetch_vulnerabilities
   ```

   并加入 `update_slow_data()` 或独立 scheduler。

4. 在 `backend/routers/data.py` 的 `/api/live-data/slow` payload 中加入：

   ```python
   "vulnerabilities": d.get("vulnerabilities", [])
   ```

5. 在 `frontend/src/types/dashboard.ts` 中增加 `VulnerabilityItem` 类型和 `DashboardData.vulnerabilities` 字段。

6. 新增前端面板或复用 News/AI Intel 面板展示。

建议先做成右侧情报列表，不急着做地图图层，因为 CVE 本身没有天然经纬度。

汉化建议：

- 保留英文原始 `title`、`summary`、`description`。
- 增加 `title_zh`、`summary_zh`、`severity_zh`、`product_name_zh` 等显示字段。
- CVE ID、CWE、CVSS、vendor/product 原始名称不要覆盖，避免影响检索和交叉引用。

### 7.3 推荐实施顺序

1. 先把 Palo Alto RSS 加入 News Feeds。
2. 在 `fetch_news()` 中增加 cyber/security 风险关键词，让安全公告不会被低估。
3. 观察 Palo Alto 源在 Threat Intel 面板中的表现。
4. 再设计 `vulnerabilities` 正式数据层，接入 CVE Services API 或 cvelistV5。
5. 最后做中文显示字段和前端漏洞情报面板。

## 8. 总结

如果目标是尽快完成可用汉化，优先处理：

1. UI 文案
2. 新闻 RSS 源
3. AI Intel 自定义中文 feed 图层
4. 静态数据中文字段

如果目标是长期维护一个中文增强版，则应保留原始字段，新增中文显示字段和前端显示映射，避免破坏外部数据匹配、测试和上游兼容性。
